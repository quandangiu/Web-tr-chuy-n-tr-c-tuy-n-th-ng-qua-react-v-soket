✨ 4. Chức năng chi tiết

👤 Authentication & User Management
  • Register/Login — bcrypt hash password, JWT access token (15 phút) + refresh token (7 ngày, httpOnly cookie)
  • Auto Refresh Token — Axios interceptor tự động gọi /auth/refresh-token khi nhận 401, queue các request pending
  • Refresh Token Revocation — Lưu trên Redis với TTL, logout xóa token khỏi Redis
  • User Profile — Avatar (Cloudinary), displayName, bio, status (online/offline/away)
  • Protected Routes — React Router wrapper kiểm tra auth state trước khi render
🏢 Workspace Management
  • CRUD Workspace — Tạo workspace tự động tạo channel #general, auto-generate slug + inviteCode
  • Invite System — Link invite /workspace/join/:inviteCode, auto-add vào public+voice channels
  • Member Roles — 3 role: Owner Admin Member
  • Admin Actions — Thay đổi role, kick member, upload workspace avatar
  • Realtime Broadcast — Mọi thay đổi workspace emit qua Socket.IO tới tất cả members
📢 Channel System
  • 4 loại channel: Public — Tất cả members tự động join Private — Chỉ người được mời DM — Tin nhắn riêng 1-1 Voice — Kênh thoại WebRTC
  • Auto-sync — Khi fetch channels, tự add user vào public+voice channels nếu chưa có
  • Unread Count — Đếm tin nhắn chưa đọc per channel
  • Encryption — Channel có thể bật AES encryption, server tạo key, encrypt/decrypt tin nhắn
  • Realtime — Tạo/sửa/xóa channel broadcast tức thì qua socket
💬 Messaging
  • Message Types: text image file system
  • File Upload — Multer → Cloudinary, hỗ trợ image/pdf/zip/docx, max 10MB
  • Reply (Quote) — Reply to message với populate sender info
  • Reactions — Emoji reactions (👍❤️😂😮😢🔥🎉👏), lưu bằng Map<emoji, userId[]>
  • Edit/Delete — Soft delete (isDeleted flag), edit tracking (isEdited + editedAt)
  • Read Receipt — readBy array, mark_read socket event batch update
  • Typing Indicator — typing_start/typing_stop events, debounce 2 giây
  • Search — MongoDB text index, full-text search trên content
  • Cursor Pagination — Infinite scroll, cursor-based (không dùng offset)
  • E2E Encryption — Khi channel bật encryption: server encrypt bằng AES trước khi lưu, decrypt khi trả về
🔊 Voice Channel (WebRTC Mesh)
  • Mesh topology — Mỗi user kết nối trực tiếp với tất cả users khác (SimplePeer)
  • State trên Redis — voice:channelId → JSON array VoiceMember[]
  • Mute/Deafen — Toggle mic (track.enabled), deafen (mute tất cả remote audio)
  • Speaking Indicator — AudioContext AnalyserNode, FFT frequency → detect speaking, emit qua socket
  • Auto-rejoin (F5) — Zustand persist session, socket reconnect → auto rejoin voice channel
  • Admin Controls — Force mute user, kick user from voice (chỉ owner/admin)
  • Stale Closure Fix — useRef mirrors cho session/currentUser, socket events register 1 lần (empty deps)
📹 Video Call (P2P)
  • Call Flow: call_user → incoming_call → accept/reject → WebRTC offer/answer/ICE
  • SimplePeer — Wrapper cho RTCPeerConnection, trickle ICE
  • ICE Servers: Google STUN + Metered TURN (TCP/UDP port 80/443)
  • Controls: Toggle camera, toggle mic, hang up
  • States: idle → calling → ringing → connected → ended
🟢 Presence System
  • Redis Key: presence:userId với TTL 5 phút
  • Heartbeat: Client gửi mỗi 4 phút, server refresh TTL
  • Status Change: Broadcast user_status_changed tới tất cả workspace rooms
  • Multi-tab: Kiểm tra còn socket nào connect trước khi set offline
  • Online Users: Redis MGET batch query khi join workspace

⚡ 5. Giải pháp Realtime

🔌 Socket.IO — Lựa chọn chính

  • Tại sao Socket.IO? — Auto-reconnect, fallback polling, room/namespace, acknowledgment callback
  • Transport: WebSocket (ưu tiên) → Long Polling (fallback)
  • Auth: JWT token trong socket.handshake.auth, middleware verify trước connection
  • Room pattern: user:userId — DM, call workspace:wsId — Presence, admin events channel:chId — Messages, typing video:roomId — Video call
  • 4 Handler modules: Presence, Message, Voice, Video

📹 6. WebRTC Deep Dive

Voice Channel — Mesh Network

  User A ◄────── WebRTC P2P ──────► User B
    ▲                                  ▲
    │          WebRTC P2P              │
    └──────────────────────────────────┘
                    │
               User C (mỗi user kết nối trực tiếp với nhau)

Signaling qua Socket.IO:
  1. User A join → server broadcast voice_user_joined
  2. Existing users tạo peer (non-initiator)
  3. New user nhận voice_channel_updated → tạo peer (initiator) cho mỗi existing user
  4. SimplePeer signal → socket.emit('voice_signal') → relay tới target
  5. ICE candidates trao đổi → P2P connection established
  6. Audio stream qua RTCPeerConnection (không qua server)

🎙 Speaking Detection

  • AudioContext → createMediaStreamSource
  • AnalyserNode FFT size 256, smoothing 0.5
  • Mỗi 100ms: tính average frequency → threshold > 8
  • Emit voice_speaking qua socket cho remote users

📹 Video Call — P2P 1-1

  • call_user → incoming_call → accept/reject
  • Initiator tạo offer → relay qua socket
  • Callee tạo answer → relay ngược lại
  • ICE candidates trickle theo real-time
  • STUN (Google) + TURN (Metered) servers

🔒 7. Bảo mật

🔑

JWT Dual Token

Access Token 15 phút (Bearer header), Refresh Token 7 ngày (httpOnly cookie, Redis whitelist)

🔐

Password Hashing

Bcrypt với salt rounds, passwordHash field select: false

🛡️

CORS Policy

Whitelist origins + regex cho LAN IPs (192.168.x.x, 10.x.x.x, 172.16-31.x.x)

🔏

Message Encryption

AES encryption per-channel, key stored server-side (select: false), encrypt/decrypt tại server

✅

Authorization

Middleware auth verify JWT. Controller-level check: member? admin? owner? per resource

📋

Validation

Zod schema validation middleware, Mongoose schema constraints (minlength, maxlength, enum, unique)

🎯 10. Câu hỏi phỏng vấn & Cách trả lời

❓ "Mô tả dự án của bạn?"

"Em đã xây dựng một nền tảng giao tiếp workspace realtime tương tự Discord/Slack, sử dụng React + Node.js. Dự án cho phép người dùng tạo workspace, mời thành viên qua invite link, chat realtime qua các channel (text/voice), gọi video P2P, và quản lý phân quyền. Điểm nổi bật là hệ thống realtime hoàn chỉnh với Socket.IO cho messaging/presence, và WebRTC cho voice/video call, cùng với message encryption AES."

❓ "Tại sao chọn Socket.IO thay vì WebSocket thuần?"

"Socket.IO cung cấp nhiều tính năng mà raw WebSocket không có: auto-reconnection (10 attempts, 2s delay), fallback transport (polling khi WS bị block), room/namespace (broadcast tới nhóm user cụ thể), acknowledgment callback (xác nhận server nhận event). Điều này giúp em xử lý các edge case như mất kết nối, multi-tab, và F5 reload một cách dễ dàng."

❓ "WebRTC hoạt động như thế nào trong dự án?"

"Em dùng WebRTC mesh network cho voice channel — mỗi user kết nối P2P trực tiếp với nhau qua SimplePeer. Signaling (trao đổi offer/answer/ICE candidates) được relay qua Socket.IO. Em dùng STUN servers (Google) để discover public IP và TURN servers (Metered) để relay khi P2P bị NAT block. Speaking detection dùng AudioContext AnalyserNode phân tích FFT frequency mỗi 100ms."

❓ "Giải thích hệ thống authentication?"

"Em implement dual token strategy: Access Token (JWT, 15 phút, gửi qua Bearer header) và Refresh Token (JWT, 7 ngày, httpOnly cookie). Refresh token được lưu trên Redis để có thể revoke. Khi access token hết hạn, Axios interceptor tự động gọi refresh endpoint, đồng thời queue các request đang pending để tránh race condition. Socket.IO cũng authenticate bằng access token trong handshake."

❓ "Redis dùng để làm gì?"

"Redis đóng 3 vai trò: (1) Presence management — key presence:userId với TTL 5 phút, heartbeat refresh; (2) Refresh token whitelist — lưu token với TTL 7 ngày, cho phép revoke; (3) Voice channel state — lưu danh sách members đang trong voice channel, cập nhật khi join/leave/mute. Redis được chọn vì tốc độ O(1) và TTL tự động cho presence."

❓ "Làm sao xử lý scalability?"
  • Cursor-based pagination thay vì offset (tránh skip overhead khi data lớn)
  • MongoDB indexes cho channel+createdAt, text search, members
  • Redis cho hot data (presence, voice state) tránh query DB
  • Cloudinary CDN cho file/image (không lưu server)
  • Docker multi-stage build, sẵn sàng horizontal scale
  • Socket.IO rooms broadcast chỉ tới users cần nhận
❓ "Khó khăn lớn nhất và cách giải quyết?"
  • Stale closure trong voice channel hooks — Socket events register 1 lần (empty deps), nhưng cần access state hiện tại → dùng useRef mirrors, subscribe Zustand store trực tiếp
  • WebRTC auto-rejoin sau F5 — Persist voice session trong Zustand, khi socket reconnect → detect persisted session → delay 1.5s (chờ auth + join_workspace) → auto rejoin
  • Multi-tab presence — Khi disconnect, kiểm tra io.in(user:id).fetchSockets() xem còn tab nào không trước khi set offline
  • Axios token refresh race condition — Queue pattern: nếu đang refresh thì queue request, khi done thì processQueue tất cả
